1

Trazabilidad con P_TRAZA_CORE

Todo procedimiento o función nuevo o modificado debe registrar trazas. El sistema permite rastrear qué pasó, cuándo, en qué paso y cuánto tardó.

¿Cómo fluye una traza?

✦
Declarar variables obligatorias
N_ID_PROCESO · N_TX · TS_INICIO — siempre las tres
I
Traza de INICIO
Inmediatamente después de inicializar N_TX. Tipo "I".
T
Trazas intermedias (opcional)
Tipo "T" — solo para hitos relevantes dentro de la lógica de negocio.
F
Traza de FIN
Antes del END exitoso. Tipo "F" — incluir P_DURACION_MS.
E
Traza de ERROR
Dentro del EXCEPTION WHEN OTHERS. Tipo "E" — incluir SQLERRM y BACKTRACE. Terminar con RAISE.

Componentes del sistema

FN_ID_PROCESO

Obtiene el ID del proceso parametrizado. Si no existe, el sistema registra "DESCONOCIDO" automáticamente.

FN_TRAE_TX

Conserva o genera el consecutivo de transacción. Recibe P_TX para propagar contexto entre subprocesos.

PR_REGISTRAR_TRAZA

Registra la traza con PRAGMA AUTONOMOUS_TRANSACTION — se graba aunque el proceso principal falle.

FN_DURACION_MS

Calcula la duración en milisegundos desde TS_INICIO hasta el momento actual.

MUSTPlantilla base completa▼
PROCEDURE PR_EJEMPLO_ESTANDAR(P_TX IN NUMBER DEFAULT NULL) IS
    N_ID_PROCESO NUMBER;
    N_TX         NUMBER;
    TS_INICIO    TIMESTAMP := SYSTIMESTAMP;
BEGIN
    N_ID_PROCESO := P_TRAZA_CORE.FN_ID_PROCESO('PAQUETE.PR_EJEMPLO_ESTANDAR');
    N_TX         := P_TRAZA_CORE.FN_TRAE_TX(P_TX);

    P_TRAZA_CORE.PR_REGISTRAR_TRAZA(
        P_ID             => N_ID_PROCESO,
        P_TIPO_EJECUCION => 'I',
        P_MENSAJE        => 'INICIO del proceso PR_EJEMPLO_ESTANDAR.',
        P_TX             => N_TX
    );

    -- Lógica de negocio aquí

    P_TRAZA_CORE.PR_REGISTRAR_TRAZA(
        P_ID             => N_ID_PROCESO,
        P_TIPO_EJECUCION => 'F',
        P_MENSAJE        => 'FIN del proceso finalizado correctamente.',
        P_TX             => N_TX,
        P_DURACION_MS    => P_TRAZA_CORE.FN_DURACION_MS(TS_INICIO)
    );
EXCEPTION
    WHEN OTHERS THEN
        P_TRAZA_CORE.PR_REGISTRAR_TRAZA(
            P_ID             => N_ID_PROCESO,
            P_TIPO_EJECUCION => 'E',
            P_MENSAJE        => 'ERROR: ' || SQLERRM ||
                                ' - BACKTRACE: ' || SYS.DBMS_UTILITY.FORMAT_ERROR_BACKTRACE,
            P_TX             => N_TX,
            P_DURACION_MS    => P_TRAZA_CORE.FN_DURACION_MS(TS_INICIO)
        );
        RAISE;
END PR_EJEMPLO_ESTANDAR;
¿Por qué PRAGMA AUTONOMOUS_TRANSACTION?
PR_REGISTRAR_TRAZA guarda la traza en una transacción independiente. Si el proceso principal hace ROLLBACK, las trazas de error igualmente quedan persistidas en la BD.

Constraints

Reglas de nombrado para restricciones de integridad en tablas Oracle.

Primary Key — PK
SHORTtabla_PK
Ej: tabla CLIENTES → CLI_PK
Foreign Key — FK
SHORTtablahija_SHORTtablapapa_FK
Ej: CUENTAS referencia CLIENTES → CTA_CLI_FK
Unique Key — UK
SHORTtabla_NOMBRE_UK
Ej: email único en CLIENTES → CLI_EMAIL_UK
Otros constraints (check, not null, etc.)
AVCON_SHORTtabla_PRIMERALETRACOLUMNA_00X
Consecutivo desde 000. Ej: AVCON_CLI_E_000 (columna ESTADO)
¿Qué es SHORTtabla? Es una abreviatura de 3–4 letras del nombre de la tabla. CLIENTES → CLI, CUENTAS → CTA, MOVIMIENTOS → MOV. Esta abreviatura se usa de forma consistente en todos los objetos relacionados a esa tabla.

1
Ubicar o crear el package
Todo código va dentro de un paquete (P_NOMBRE). Nunca suelto.
Regla clave: El nombre del paquete sigue el patrón P_NOMBRE. Siempre tiene spec (interfaz pública) y body (implementación) en archivos separados.

Spec del paquete (archivo: P_CLIENTES.pks):

-- Objetivo: Gestión de clientes del sistema
CREATE OR REPLACE PACKAGE P_CLIENTES AS

    -- Actualiza el estado de un cliente
    PROCEDURE PR_ACTUALIZAR_ESTADO(
        P_CLIENTE_ID IN CLIENTES.CLIENTE_ID%TYPE,
        P_ESTADO     IN CLIENTES.ESTADO%TYPE,
        P_TX         IN NUMBER DEFAULT NULL
    );

END P_CLIENTES;
2
Encabezado del body + firma del procedimiento
Incluir siempre P_TX IN NUMBER DEFAULT NULL al final de los parámetros.
P_TX es obligatorio en todos los procedimientos. Permite que el llamador propague su contexto de transacción. El DEFAULT NULL hace que sea opcional cuando se llama externamente.

Body del paquete (archivo: P_CLIENTES.pkb):

-- Objetivo: Gestión de clientes del sistema
CREATE OR REPLACE PACKAGE BODY P_CLIENTES AS

    -- Actualiza el estado de un cliente
    PROCEDURE PR_ACTUALIZAR_ESTADO(
        P_CLIENTE_ID IN CLIENTES.CLIENTE_ID%TYPE,
        P_ESTADO     IN CLIENTES.ESTADO%TYPE,
        P_TX         IN NUMBER DEFAULT NULL  -- SIEMPRE al final
    ) IS
        -- variables van aquí (paso 3)
    BEGIN
        NULL; -- lógica va aquí (pasos 4-7)
    END PR_ACTUALIZAR_ESTADO;

END P_CLIENTES;
3
Sección declarativa en orden correcto
Cursores → Registros → Colecciones → Constantes → Escalares → Trazabilidad → Excepciones.
Las variables de trazabilidad (N_ID_PROCESO, N_TX, TS_INICIO) van siempre al final de la sección declarativa, en posición 6. TS_INICIO se inicializa inline con SYSTIMESTAMP.
    PROCEDURE PR_ACTUALIZAR_ESTADO(
        P_CLIENTE_ID IN CLIENTES.CLIENTE_ID%TYPE,
        P_ESTADO     IN CLIENTES.ESTADO%TYPE,
        P_TX         IN NUMBER DEFAULT NULL
    ) IS
        -- 4. Constantes
        C_MODULO CONSTANT VARCHAR2(100) := 'P_CLIENTES.PR_ACTUALIZAR_ESTADO';

        -- 5. Variables escalares
        V_ESTADO_ACTUAL CLIENTES.ESTADO%TYPE;
        B_EXISTE        BOOLEAN := FALSE;

        -- 6. Variables de trazabilidad (SIEMPRE las tres)
        N_ID_PROCESO NUMBER;
        N_TX         NUMBER;
        TS_INICIO    TIMESTAMP := SYSTIMESTAMP;

        -- 7. Excepciones de negocio
        E_CLIENTE_INVALIDO EXCEPTION;
    BEGIN
        NULL;
    END PR_ACTUALIZAR_ESTADO;
4
Inicializar N_ID_PROCESO y N_TX
Primeras dos líneas del BEGIN, siempre en este orden.
FN_ID_PROCESO recibe el nombre completo 'PAQUETE.PROCEDIMIENTO' — esto es lo que aparece en las trazas. Si no está parametrizado en la tabla de procesos, el sistema registra "DESCONOCIDO" pero no falla.

FN_TRAE_TX recibe P_TX. Si viene NULL (llamada externa), genera un nuevo consecutivo. Si viene con valor, lo conserva para que todas las subtrazas queden bajo el mismo contexto.
    BEGIN
        -- Inicialización de trazabilidad (SIEMPRE primero)
        N_ID_PROCESO := P_TRAZA_CORE.FN_ID_PROCESO('P_CLIENTES.PR_ACTUALIZAR_ESTADO');
        N_TX         := P_TRAZA_CORE.FN_TRAE_TX(P_TX);

        -- Lógica de negocio viene aquí...
    END PR_ACTUALIZAR_ESTADO;
5
Registrar traza de INICIO (tipo "I")
Inmediatamente después de inicializar N_TX. Sin duración.
La traza "I" no lleva P_DURACION_MS — apenas está comenzando. El mensaje debe decir claramente qué proceso inicia. Puede incluir parámetros clave para facilitar diagnóstico.
    BEGIN
        N_ID_PROCESO := P_TRAZA_CORE.FN_ID_PROCESO('P_CLIENTES.PR_ACTUALIZAR_ESTADO');
        N_TX         := P_TRAZA_CORE.FN_TRAE_TX(P_TX);

        -- Traza de inicio
        P_TRAZA_CORE.PR_REGISTRAR_TRAZA(
            P_ID             => N_ID_PROCESO,
            P_TIPO_EJECUCION => 'I',
            P_MENSAJE        => 'INICIO PR_ACTUALIZAR_ESTADO. Cliente: ' || P_CLIENTE_ID,
            P_TX             => N_TX
        );

        -- Lógica de negocio...

    END PR_ACTUALIZAR_ESTADO;
6
Lógica de negocio + traza de FIN (tipo "F")
La traza F siempre lleva P_DURACION_MS calculada desde TS_INICIO.
Traza "T" para hitos intermedios: Si la lógica tiene pasos relevantes (ej: "procesados 500 registros", "validación exitosa"), se puede agregar una traza tipo "T" entre la "I" y la "F" para mayor granularidad en el monitoreo.
        -- Lógica de negocio
        SELECT ESTADO INTO V_ESTADO_ACTUAL
        FROM   CLIENTES
        WHERE  CLIENTE_ID = P_CLIENTE_ID;

        IF V_ESTADO_ACTUAL = P_ESTADO THEN
            RAISE E_CLIENTE_INVALIDO;
        END IF;

        UPDATE CLIENTES
        SET    ESTADO = P_ESTADO
        WHERE  CLIENTE_ID = P_CLIENTE_ID;

        -- Traza de FIN (con duración)
        P_TRAZA_CORE.PR_REGISTRAR_TRAZA(
            P_ID             => N_ID_PROCESO,
            P_TIPO_EJECUCION => 'F',
            P_MENSAJE        => 'FIN PR_ACTUALIZAR_ESTADO. Estado actualizado a: ' || P_ESTADO,
            P_TX             => N_TX,
            P_DURACION_MS    => P_TRAZA_CORE.FN_DURACION_MS(TS_INICIO)  -- obligatorio en F
        );

    END PR_ACTUALIZAR_ESTADO;
7
Bloque EXCEPTION con traza de ERROR (tipo "E")
Siempre capturar SQLERRM + BACKTRACE. Siempre terminar con RAISE.
RAISE al final del WHEN OTHERS es obligatorio. Sin él, el error queda "tragado" y el llamador nunca sabe que algo falló. El RAISE propaga el error hacia arriba en la cadena de llamadas.
EXCEPTION
        -- Excepciones de negocio propias (opcional si aplica)
        WHEN E_CLIENTE_INVALIDO THEN
            P_TRAZA_CORE.PR_REGISTRAR_TRAZA(
                P_ID             => N_ID_PROCESO,
                P_TIPO_EJECUCION => 'E',
                P_MENSAJE        => 'Cliente ya tiene ese estado.',
                P_TX             => N_TX,
                P_DURACION_MS    => P_TRAZA_CORE.FN_DURACION_MS(TS_INICIO)
            );
            RAISE;

        -- Captura general (SIEMPRE presente)
        WHEN OTHERS THEN
            P_TRAZA_CORE.PR_REGISTRAR_TRAZA(
                P_ID             => N_ID_PROCESO,
                P_TIPO_EJECUCION => 'E',
                P_MENSAJE        => 'ERROR en PR_ACTUALIZAR_ESTADO. SQLERRM: ' || SQLERRM ||
                                    ' - BACKTRACE: ' || SYS.DBMS_UTILITY.FORMAT_ERROR_BACKTRACE,
                P_TX             => N_TX,
                P_DURACION_MS    => P_TRAZA_CORE.FN_DURACION_MS(TS_INICIO)
            );
            RAISE;  -- SIEMPRE propagar
    END PR_ACTUALIZAR_ESTADO;
¡Procedimiento completo! Ya tienes firma correcta, sección declarativa en orden, inicialización de trazabilidad, traza I, lógica de negocio, traza F y bloque EXCEPTION con traza E + RAISE.

¿Qué considerar en el proyecto?

Aspectos prácticos y contexto que debes tener en cuenta al aplicar este estándar en los paquetes existentes del proyecto.

1
Definir los SHORTs de cada tabla antes de crear objetos

El SHORT es una abreviatura de 3–4 letras que identifica cada tabla. Debe estar definido y acordado en el equipo antes de crear constraints, índices o secuencias, porque cambiar el SHORT después implica renombrar todos los objetos dependientes.

CLIENTESCLICLI_PK
CUENTASCTACTA_PK
MOVIMIENTOSMOVMOV_PK
TRANSACCIONESTRXTRX_PK
PARAMETROSPARPAR_PK

2
Registrar el proceso en P_TRAZA_CORE antes de ejecutar

FN_ID_PROCESO busca el nombre del procedimiento en una tabla de procesos parametrizados. Si no lo encuentra, funciona igual pero registra el proceso como "DESCONOCIDO" en las trazas — esto dificulta el monitoreo y las alertas automáticas.

¿Qué hacer? Antes de poner en producción un nuevo procedimiento, registrar el proceso en la tabla de parámetros de P_TRAZA_CORE con el nombre exacto que se usa en FN_ID_PROCESO: 
INSERT INTO T_PROCESOS_TRAZA (NOMBRE_PROCESO, DESCRIPCION)
VALUES ('P_CLIENTES.PR_ACTUALIZAR_ESTADO', 'Actualiza el estado de un cliente');

3
Código legacy sin trazas: qué hacer al modificarlo

El estándar aplica a todo código nuevo o modificado. Cuando debas tocar un procedimiento legacy que no tiene trazas, aprovecha ese momento para agregarlas siguiendo la plantilla. No es necesario refactorizar código que no vas a tocar.
Código legacy (sin modificar)
-- OK dejar así si no se toca
PROCEDURE PR_VIEJO IS
BEGIN
    UPDATE CLIENTES SET ...;
END PR_VIEJO;
Al modificar → agregar estándar
PROCEDURE PR_VIEJO(
    P_TX IN NUMBER DEFAULT NULL
) IS
    N_ID_PROCESO NUMBER;
    N_TX         NUMBER;
    TS_INICIO    TIMESTAMP := SYSTIMESTAMP;
BEGIN
    -- init + traza I + lógica + F
END PR_VIEJO;

4
Cómo propagar N_TX en cadenas de llamadas

Cuando un procedimiento A llama a un subprocedimiento B, debe pasarle N_TX. Así todas las trazas de esa operación quedan bajo el mismo consecutivo de transacción y se pueden filtrar juntas en el monitoreo. 
-- En PR_PROCESAR_PEDIDO llama subprocedimientos:
P_INVENTARIO.PR_RESERVAR_STOCK(
    P_PRODUCTO_ID => N_PRODUCTO,
    P_CANTIDAD    => N_CANTIDAD,
    P_TX          => N_TX  -- <-- propagar siempre
);

P_FACTURACION.PR_GENERAR_FACTURA(
    P_PEDIDO_ID => N_PEDIDO,
    P_TX        => N_TX  -- <-- mismo N_TX
);
En el log de trazas podrás filtrar por N_TX y verás el INICIO del padre, luego los INICIO/FIN de cada hijo, y el FIN del padre — todo en orden cronológico bajo el mismo identificador.

5
¿Cuándo usar traza "T" (hito intermedio)?

La traza "T" es opcional pero muy útil en procedimientos que procesan grandes volúmenes o tienen etapas bien diferenciadas. Agrega uno cuando quieras saber exactamente en qué paso estaba el proceso si algo falla a mitad. 
-- Ejemplo en un proceso de carga masiva:
P_TRAZA_CORE.PR_REGISTRAR_TRAZA(
    P_ID             => N_ID_PROCESO,
    P_TIPO_EJECUCION => 'T',
    P_MENSAJE        => 'Validación completada. Registros válidos: ' || N_VALIDOS ||
                        '. Inválidos: ' || N_INVALIDOS,
    P_TX             => N_TX,
    P_DURACION_MS    => P_TRAZA_CORE.FN_DURACION_MS(TS_INICIO)
);

Errores comunes y cómo evitarlos

Olvidar RAISE en el EXCEPTION
El error queda "tragado". El llamador recibe éxito aunque el proceso falló internamente. Dificulta debugging y produce datos inconsistentes.
No pasar P_TX a subprocedimientos
Cada subproceso genera su propio N_TX. Las trazas quedan dispersas y es imposible reconstruir la cadena completa de ejecución.
Poner DEFAULT en columnas de tabla
El valor se asigna invisible en BD. Si cambia el requerimiento, hay que buscarlo en el DDL de la tabla, no en la lógica de negocio donde debería estar.
Variables genéricas: V1, AUX, TMP
Después de 3 meses nadie (incluyendo tú mismo) sabe qué hace esa variable. Usar nombres funcionales: N_SALDO_DISPONIBLE, V_CODIGO_RESPUESTA.
FK sin índice
Oracle no crea índices automáticos sobre FK. Un DELETE de un cliente sin índice en CUENTAS.CLI_CONSECUTIVO hace full scan de toda la tabla.
Código comentado en el body
El código comentado genera confusión sobre si está activo o no. Si se eliminó con intención, usar Git para recuperarlo si hace falta.

Correcciones rápidas

Al crear una nueva tabla
1. Definir SHORT · 2. PK con NUMBER(10) · 3. Comentarios de columnas · 4. Índice para FK · 5. Índice para columnas fecha · 6. Secuencia SHORT_SEQ
Al crear un nuevo procedimiento
1. Ubicarlo en el pkg correcto · 2. Agregar P_TX al final · 3. Sección declarativa en orden · 4. Init trazabilidad · 5. Traza I + F + E
Al revisar código de otro
Verificar: ¿tiene P_TX? ¿Tiene las 3 variables de traza? ¿Tiene traza I, F y E? ¿El EXCEPTION termina en RAISE? ¿Usa notación nombrada?
